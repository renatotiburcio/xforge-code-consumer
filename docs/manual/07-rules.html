07 - Rules (41 regras comportamentais)

O que e

Rules sao regras comportamentais carregadas no system prompt do LLM. XForge tem 41 rules em 5 niveis de precedencia. Diferente de skills (referencia) ou commands (workflows), rules sao restricoes: violacoes devem ser explicadas ou corrigidas antes de prosseguir.

Quando usar

  • Quando voce precisa entender a precedencia (GCF > Golden > Security > Quality > Operational)
  • Quando quer criar rule custom para sua equipe
  • Quando uma rule e violada e voce precisa do racional
  • Quando quer auditar rules desatualizadas

Como usar

Sintaxe canonica (rules sao carregadas automaticamente):

# Rules sao carregadas no system prompt
# Para ver regras ativas:
Get-ChildItem .kilo/rules/

# Ler regra especifica
cat .kilo/rules/02-genius-council-framework.md

# Criar rule nova
cat > .kilo/rules/minha-rule.md <<EOF
---
id: minha-rule
priority: medium
applicabilityScope: ["dotnet"]
status: approved
---
# Minha Rule
[conteudo...]
EOF

regenerate-registries.ps1

Parametros

ParametroTipoDefaultDescricao
idstringobrigatorioIdentificador unico. kebab-case en.
prioritystringmediumcritical, high, medium, low.
applicabilityScopearray["*"]Stacks compativeis. Router filtra.
statusstringdraftdraft, approved, deprecated.

Exemplos praticos

Cenario 1: Precedencia (5 niveis)

Contexto: Voce tem duvida sobre qual regra vence em conflito.

# Hierarquia canonica
1. GCF (Regra de Ouro Suprema)
   - .kilo/rules/02-genius-council-framework.md
2. Golden Rules (stack-agnostic)
   - .kilo/rules/01-xforge-golden-rules.md (SOLID, pacotes estaveis, stack-agnostic)
3. Governanca (RBAC, security, LGPD, audit)
   - .kilo/rules/policy-rules.md, rbac-rules.md, security-rules.md
4. Qualidade (gates, loop, scoring)
   - .kilo/rules/proactive-quality-gates.md, loop-discipline.md
5. Memoria & Conhecimento
   - .kilo/rules/session-memory.md, knowledge-rules.md

# Conflito: Golden Rule diz "use AutoMapper", Skill diz "use XForge.MediatR"
# Resolucao: Golden Rule vence (N2 > N4)
# Mas: Skill canonica XForge.MediatR eh excecao documentada (golden-rules.md)

Output esperado:

[rule-precedence] Conflito detectado: golden-rules vs xforge-mediatr
[resolution] GCF N1 + Golden N2 + XForge.MediatR (skill canonica)
[verdict] AutoMapper para mapping, XForge.MediatR para CQRS. Nao confundir.

Edge case: Em caso de empate, DR recente vence. Se conflito nao resolvido, acione GCF.

Cenario 2: Golden Rules (stack-agnostic)

Contexto: Voce quer entender o que NUNCA quebrar em qualquer stack.

cat .kilo/rules/01-xforge-golden-rules.md

# Principais regras
# 1. Stack-agnostic (DR-0180) - XForge NAO assume stack
# 2. SOLID + SRP (todos stacks)
# 3. Pacotes estaveis (NUNCA preview, beta, rc, alpha, nightly)
# 4. Clean Code
# 5. Documentacao (README 5 min, XML comments, OpenAPI)
# 6. Diagramas C4 L1/L2/L3 para sistemas > 3 modulos

Output esperado (resumo da rule):

## Regra de ouro 0 - Stack-agnostic
- Detectar stack antes de sugerir patterns
- Perguntar quando ambiguo (max 3 opcoes)
- Diretores especialistas sao OPT-IN por stack
- Comandos genericos suportam TODOS os stacks

## Regra de ouro 1 - SOLID e SRP
- Cada arquivo: uma unica responsabilidade
- Services gigantes PROIBIDOS
- Handlers com multiplas responsabilidades PROIBIDOS

Edge case: Golden Rules sao inviolaveis. Violacao => GCF obrigatorio + DR formal.

Cenario 3: Self-Healing Rules (SH-001 a SH-012)

Contexto: Voce quer entender correcoes automaticas pre-commit.

# 12 regras self-healing
SH-001: Unused imports (remove)
SH-002: Missing Async suffix (adiciona)
SH-003: Null-conditional access (adiciona ?.)
SH-004: Missing null check (adiciona if null return)
SH-005: Async over sync (substitui .Result por await)
SH-006: Missing using declaration (adiciona await using)
SH-007: Parameterized query (security: SQL injection)
SH-008: Secret detection (move para .env)
SH-009: CancellationToken (adiciona param)
SH-010: IHttpClientFactory (substitui new HttpClient)
SH-011: ConfigureAwait(false) (lib/service)
SH-012: Disposable pattern (GC.SuppressFinalize)

Output esperado (exemplo SH-008):

## SH-008: Secret Detection
- Detecta: API key, password, token em codigo
- Acao: Move para variavel de ambiente ou appsettings.json
- Severidade: critical (security)
- Auto-fix: parcial (move, nao substitui valor)

# Exemplo
// ANTES (detectado):
var apiKey = "sk-1234567890abcdef";

// DEPOIS (auto-fix):
var apiKey = _configuration["ExternalServices:ApiKey"]
    ?? throw new InvalidOperationException("ApiKey not configured");

Edge case: SH-007 (SQL injection) e SH-008 (secrets) sao critical e SEMPRE aplicados.

Cenario 4: Quality Gates (proactive)

Contexto: Voce quer entender checks automaticos stack-aware.

cat .kilo/rules/proactive-quality-gates.md

# Stack detection (15 sinais)
*.csproj -> dotnet
package.json -> node
pyproject.toml -> python
go.mod -> go
Cargo.toml -> rust
angular.json -> angular
next.config.* -> next
...

# Gates
Pre-Commit: format + secrets + self-healing (< 2s)
Pre-Build: restore + build + warn-as-error (< 30s)
Pre-Push: test + coverage 85% + security (< 60s)
Post-Merge: full build + test + audit (< 5min)

# .NET specific
dotnet format --verify-no-changes
dotnet test --collect:"XPlat Code Coverage"
dotnet list package --vulnerable

Output esperado (gate pre-commit):

[pre-commit] dotnet format --verify-no-changes
[pre-commit] gitleaks detect
[pre-commit] SH rules (auto-fix)
[done] < 2s, commit allowed

Edge case: Em monorepo, gates rodam em cada stack. CI pode ser lento.

Cenario 5: Criar rule custom para equipe

Contexto: Sua equipe quer rule especifica (ex: sempre usar Xunit).

cat > .kilo/rules/minha-equipe-xunit.md <<EOF
---
id: minha-equipe-xunit
priority: medium
applicabilityScope: ["dotnet"]
status: approved
---
# Xunit como padrao
Sempre use Xunit (NUNCA NUnit ou MSTest).
Use FluentAssertions para asserts.
Use Moq para mocks.
Use Testcontainers para integracao.
Cobertura minima: 85% (linhas), 80% (branches).
EOF

regenerate-registries.ps1
doctor.ps1

Output esperado:

[regenerate] 45 rules (era 44)
[doctor] 0 errors
[done] Rule custom ativa em projetos .NET

Edge case: Rules sao globais para o template. Para regras por-projeto, use .kilo/.stack-context.json overrides.

Troubleshooting

Rule nao carrega

Causa: applicabilityScope nao bate com stack.

Solucao: stack-detector.ps1 --check-scope .kilo/rules/ para ver quais carregam.

Conflito entre rules

Causa: Multiplas rules cobrem o mesmo topico.

Solucao: Hierarquia canonica (5 niveis). Em empate, DR recente vence. Se ainda ambiguo, acione GCF.

Rule impede acao necessaria

Causa: Rule muito restritiva para seu caso.

Solucao: Documente excecao em DR formal. Aicione ao applicabilityScope se a rule for especifica demais. Ou use --no-verify em commit (emergencias).

Rule deprecated mas ainda carrega

Causa: Frontmatter nao foi atualizado para status: deprecated.

Solucao: Edite frontmatter. Rules deprecated sao ignoradas pelo router mas ainda existem no filesystem.
