Cat-skills Arquitetura - DDD, Clean, Performance

O que e

Catalogo de skills para design e arquitetura de software. Inclui: Domain-Driven Design (DDD - bounded contexts, aggregates, entities, value objects), Clean Architecture (entities, use cases, interface adapters, frameworks), Hexagonal Architecture (ports & adapters), CQRS (Command Query Responsibility Segregation), Event Sourcing, Saga Pattern, Microservices vs Monolith, performance (profiling, caching, indexing), scalability (horizontal/vertical), patterns GoF, SOLID, GRASP. Skills canonicas: architecture-enterprise-expert, csharp-clean-code-expert, dotnet-standards, software-development-expert.

Quando usar

  • Quando precisa desenhar um sistema novo do zero
  • Quando precisa refatorar um monolito em microservicos
  • Quando precisa melhorar performance (profiling + caching)
  • Quando precisa modelar dominio complexo (DDD)
  • Quando precisa revisar arquitetura existente

Como usar

Skills arquitetura funcionam como consultores: fazem perguntas sobre o dominio, sugerem patterns, identificam code smells. Use /arquitetura-consultar para diagnostico inicial.

# Consultoria inicial
/arquitetura-consultar --projeto 
# Aplicar DDD bounded context
/ddd-bounded-context --domain 'Vendas' --entities Pedido,Cliente,Produto
# Analise de performance
/performance-profile --app  --duration 30s

Parametros

ParametroTipoDefaultDescricao
--patternstringdddPattern a aplicar: ddd, clean, hexagonal, cqrs, event-sourcing, saga
--projetopathatualPath do projeto a analisar
--scopestringallEscopo: entities, use-cases, interfaces, infrastructure, all
--metricasstringperformanceMetricas: performance, memory, latency, throughput

Exemplos praticos

Cenario 1: Aplicar DDD em projeto novo

Contexto: Voce esta comecando ERP e precisa modelar o dominio.

/ddd-bounded-context --domain 'ERP' --subdomains 'Vendas,Estoque,Financeiro,Fiscal'

Output esperado:

4 bounded contexts identificados. Cada um com: aggregate root, entities, value objects, domain events. Mapeamento de contextos (Context Map).

Edge case: DDD nao e obrigatorio para todo projeto - use para dominios complexos. Para CRUD simples, KISS.

Cenario 2: Refatorar monolito para Clean Architecture

Contexto: Voce tem projeto .NET antigo e quer separar camadas.

/clean-architecture-refactor --projeto  --camadas 'Domain,Application,Infrastructure,API'

Output esperado:

Estrutura criada: src/Domain (entities, value objects), src/Application (use cases, services), src/Infrastructure (DB, external APIs), src/API (controllers). Dependencias apontam para Domain.

Edge case: NUNCA referencie Infrastructure de Domain. Domain deve ser pure C# (sem dependencias externas, exceto interfaces).

Cenario 3: Profiling de performance

Contexto: Sua API esta lenta. Quer identificar o gargalo.

/performance-profile --app MyAPI --duration 30s --samples 100

Output esperado:

CPU: 45% gasto em JsonSerializer.Deserialize (18%), DB query X (12%), HTTP client Y (8%). Alocacoes: 12 MB/s. Top 5 metodos gastam 73% do tempo.

Edge case: Profile em ambiente REPRESENTATIVO (producao ou staging). Profile em dev pode ser enganoso (dados, configuracoes).

Cenario 4: Cache com Redis

Contexto: API tem endpoint GET /produtos chamado 1000x/s, banco sobrecarregado.

/cache-strategy --endpoint /produtos --ttl 300 --invalidacao manual

Output esperado:

Cache Redis adicionado (StackExchange.Redis). Key: 'produtos:all'. TTL 300s. Invalidacao: via pub/sub quando produto e criado/atualizado. Cache hit ratio esperado: 85%.

Edge case: Para dados altamente dinamicos, considere cache write-through (atualiza cache junto com DB). Para read-heavy estatico, cache simple TTL funciona.

Cenario 5: CQRS com Event Sourcing

Contexto: Voce precisa de auditoria completa de mudancas (banco, fiscal, compliance).

/cqrs-event-sourcing --aggregates Pedido,Cliente --events 'PedidoCriado,PedidoPago,PedidoEnviado'

Output esperado:

Event Store (tabela eventos) + Command side (write) + Query side (read projections). Cada mudanca e um evento imutavel. Replay possivel.

Edge case: Event Sourcing e complexo - use so quando auditoria completa e necessaria. Para 90% dos casos, log de mudancas + auditoria simples basta.

Troubleshooting

Onde coloco a validacao (Domain ou Application)?

Causa: Regra do DDD: validacao de invariantes do agregado vai no Domain. Validacao de input vai na Application (DTOs).

Solucao: Se validacao e regra de negocio (ex: pedido nao pode ter mais de 100 itens), vai no Domain (no entity ou value object). Se e validacao tecnica (ex: campo obrigatorio, formato), vai na Application (com FluentValidation).

Quando usar Clean Architecture vs DDD vs Hexagonal?

Causa: Sao complementares, nao excludentes

Solucao: Clean = separacao de camadas (dependencias). DDD = modelagem do dominio. Hexagonal = isolacao do dominio via ports/adapters. Combine: Clean Architecture + DDD + Hexagonal ports = ideal para projetos complexos.

Performance ruim mesmo com Clean Architecture

Causa: Clean Architecture adiciona indirecao (interfaces, adapters), pode ter overhead

Solucao: Para microservicos de alta performance, simplifique. Para dominios complexos, mantenha. Meça com profiler antes de otimizar prematuramente.

CQRS adiciona complexidade desnecessaria

Causa: CQRS resolve problemas especificos (read vs write muito diferentes, event sourcing). Nem todo projeto precisa.

Solucao: Comece com CRUD simples. Adicione CQRS quando: read e write tem modelos muito diferentes, ou precisa de auditoria completa, ou precisa escalar reads independentemente.
