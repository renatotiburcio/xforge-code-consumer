14 - RAG System (Retrieval-Augmented Generation)

O que e

RAG e' o sistema de Retrieval-Augmented Generation do XForge: indexa 747 documentos locais (knowledge, skills, agents, commands, rules) em indice lexical inverted + chunks jsonl, e permite busca por termos com snippets relevantes. Diferente de RAG vector (que precisa de embeddings e GPU), este e' 100% local, sem GPU, sem cloud.

Localizacao: .xforge/rag/ (manifest.json + indexes/lexical.json + chunks/chunks.jsonl + reports/) + .xforge/scripts/rag_local.py (indexer). Config em .xforge/rag/config.json.

Quando usar

  • Quando voce precisa encontrar uma regra, skill, knowledge ou DR sem saber o nome exato
  • Quando voce quer ver contexto ao redor de um termo (10-15 termos vizinhos)
  • Quando voce precisa auditar o que esta no indice (status, cobertura, drift)
  • Quando o pre-push hook avisa RAG stale (148 docs modificados)
  • NAO use para busca semantica (use RAG vector, nao implementado por padrao)
  • NAO use para busca em arquivos fora de config.sources (apenas 9 paths indexados)

Como usar

Sintaxe canonica:

python .xforge/scripts/rag_local.py {reindex|status}

Para uso direto em queries (read-only):

# Status atual do indice
python .xforge/scripts/rag_local.py status

# Reindex completo (atualiza manifest + chunks + lexical)
python .xforge/scripts/rag_local.py reindex

# Query em Python (via shell)
python -c "import json; d=json.load(open('.xforge/rag/indexes/lexical.json')); print([c for c in d['terms'].get('dotnet', [])][:5])"

Parametros

ParametroTipoDefaultDescricao
commandstringobrigatorioreindex ou status.
--path <file>stringconfig.sourcesOverride: escanear path especifico (em breve).
--max-lines <N>int40Tamanho maximo do chunk em linhas. Maior = mais contexto, menos granular.
--overlap <N>int5Linhas de overlap entre chunks. Maior = melhor recall, mais espaco.
--quietboolfalseSuprime output, retorna exit code apenas.
--jsonboolfalseOutput em JSON em vez de human-readable.

Exemplos praticos

Cenario 1: Status diario do RAG

Contexto: Voce comeca o dia, quer saber se o RAG esta atualizado. Comando rapido.

cd meu-projeto
python .xforge/scripts/rag_local.py status

Output esperado:

{
  "lastIndexedAt": "2026-06-21T19:00:22.255841+00:00",
  "documentCount": 747,
  "chunkCount": 2529,
  "stale": false,
  "staleCount": 0
}

Edge case: Se stale: true, rode reindex. Se lastIndexedAt e' null, manifest nao existe - rode reindex.

Cenario 2: Reindex completo apos muitas mudancas

Contexto: Voce modificou 148 arquivos (skills novas, knowledge entries, etc). Pre-push hook avisou stale. Reindex agora.

python .xforge/scripts/rag_local.py reindex

Output esperado:

[reindex] Sources: 9
[reindex] Excludes: 17 patterns
[reindex] Chunk config: 40 lines, overlap 5
[reindex]   .xforge/knowledge: 295 files
[reindex]   .kilo/commands: 155 files
[reindex]   .kilo/skills: 172 files
[reindex]   .kilo/agents: 80 files
[reindex]   .kilo/rules: 44 files
[reindex]   AGENTS.md: 1 files
[reindex]   README.md: 1 files
[reindex] DONE: 747 docs, 2529 chunks, 17042 terms

Edge case: Reindex demora 5-15 segundos para 747 docs. Para projetos com 10k+ docs, use --max-lines 60 --overlap 10 para reduzir chunks.

Cenario 3: Reindex com chunk customizado (docs grandes)

Contexto: Voce tem knowledge entries grandes (200+ linhas). Chunk padrao de 40 linhas gera muito overlap. Ajuste.

python .xforge/scripts/rag_local.py reindex --max-lines 80 --overlap 10

Output esperado:

[reindex] Chunk config: 80 lines, overlap 10
[reindex] DONE: 747 docs, 1502 chunks, 16900 terms
# Chunks reduziram de 2529 -> 1502 (menor granularidade, mais espaco por chunk)

Edge case: Chunks muito grandes (>200 linhas) perdem granularidade. Para docs enormes (>500 linhas), divida em arquivos menores antes de reindexar.

Cenario 4: Configurar sources customizados

Contexto: Voce tem documentacao em ./docs-empresa/ que quer indexar junto. Adiciona source no config.

# Editar .xforge/rag/config.json
{
  "sources": [
    ".xforge/knowledge",
    ".xforge/engineer",
    ".kilo/commands",
    ".kilo/skills",
    ".kilo/agents",
    ".kilo/rules",
    "AGENTS.md",
    "README.md",
    "./docs-empresa"  # NOVO
  ]
}
# Rodar reindex
python .xforge/scripts/rag_local.py reindex

Output esperado:

[reindex]   ./docs-empresa: 23 files
[reindex] DONE: 770 docs, 2605 chunks, 17421 terms

Edge case: Paths com espacos precisam de aspas. Paths absolutos funcionam mas perdem portabilidade (use relativo). Adicionar fonte nao-indexada dispara warning mas nao aborta.

Cenario 5: Pre-push hook automatico (ja integrado)

Contexto: Voce quer que o RAG reindex rode automaticamente antes de cada push. O template ja tem isso em .git/hooks/pre-push (verifique se ativo).

git push origin master
# Pre-push hook:
#   [RAG health] stale check
#   if stale > 0:
#     warn: "Reindex recommended"
#     # NAO bloqueia push (apenas avisa)
#   else:
#     ok

Output esperado (se stale):

[RAG health] stale=3, added=[...], changed=[...]
WARN: RAG health check failed. Reindex recommended.
Pre-push checks complete.
git push ... (passa com warning)

Edge case: O hook nao bloqueia push. Se voce quer forcar reindex automatico, edite o hook para chamar python .xforge/scripts/rag_local.py reindex antes do stale check.

Troubleshooting

"RAG stale" aparece sempre, mesmo apos reindex

Causa: Algum arquivo foi modificado entre o reindex e o pre-push (race condition).

Solucao: Rode reindex novamente imediatamente antes do push. Ou use --quiet em CI para que o hook apenas reporte.

Query retorna 0 resultados para termo que existe

Causa: Termo nao foi tokenizado. Stopword, ou menos de 3 chars, ou pontuacao.

Solucao: RAG tokeniza apenas tokens com 3+ chars alfanumericos. "API" funciona. "AI" nao (2 chars). "node.js" vira "node" e "js" separados.

Reindex demora muito (>5 min)

Causa: Muitos arquivos (>5k) ou chunks muito pequenos (overhead de I/O).

Solucao: --max-lines 80 --overlap 10 para reduzir chunks. Adicione excludes em config.excludePatterns para pastas irrelevantes (ex: "node_modules", "dist", "coverage").

"chunks.jsonl corrompido" / JSON parse error

Causa: Reindex foi interrompido (Ctrl+C, OOM, disk full).

Solucao: Delete .xforge/rag/chunks/chunks.jsonl e rode reindex novamente. Reindex e' idempotente: rebuilda tudo a partir dos sources.
