15 - Quality Gates (Stack-Aware)

O que e

Quality Gates sao scripts automaticos stack-aware que validam build, testes, cobertura, encoding, seguranca, e canonical structures antes/depois de commits e pushes. Detectam o stack do projeto (.NET, Node, Python, etc) e rodam os gates apropriados. Diferente de CI tradicional (que so roda em push), XForge valida localmente em cada commit.

Localizacao: .kilo/automation/scripts/ (90+ scripts PS1/Python). Orquestrador: .kilo/automation/scripts/doctor.ps1 (executa todos os gates em sequencia).

Quando usar

  • Antes de git commit (gate pre-commit, automatico via hook)
  • Antes de git push (gate pre-push, automatico via hook)
  • Em CI/CD (rode doctor.ps1 no pipeline)
  • Para auditar estado atual sem commit (rode doctor.ps1 direto)
  • Quando quer validar um gate especifico (ex: so encoding, so RAG)
  • NAO use para rodar testes unitarios de codigo de producao (gate do template nao roda testes do projeto, so do proprio template)

Como usar

Sintaxe canonica:

.\.kilo\automation\scripts\doctor.ps1                    # Roda todos os gates
.\.kilo\automation\scripts\check-manual.ps1              # Gate 4: manual canonico DR-0181
.\.kilo\automation\scripts\check-manual-content.ps1      # Gate 5: conteudo DR-0182 (B-090)
.\.kilo\automation\scripts\check_loop_discipline.py      # Valida 6/6 PASS do loop
.\.kilo\automation\scripts\knowledge-validation.ps1       # Valida INDEX.json
.\.kilo\automation\scripts\regenerate-registries.ps1     # Regenera registries do filesystem

Para gates pre-commit/push, eles rodam via git hooks (.git/hooks/pre-commit, pre-push).

Parametros

ParametroTipoDefaultDescricao
--verboseboolfalseOutput detalhado, cada check individual.
--quietboolfalseSuprime output, retorna exit code (0/1).
--gate <N>intallRoda so o gate N (1 a 5).
--no-colorboolfalseDesativa cores (para log em CI).
--jsonboolfalseOutput em JSON para parsing em pipelines.
--fixboolfalseAplica auto-fix quando possivel (ex: regenerate-registries).
--dry-runboolfalseMostra o que faria sem executar.

Exemplos praticos

Cenario 1: Validar estado local (pre-commit manual)

Contexto: Voce modificou varios arquivos e quer ver se passam todos os gates antes de commitar.

doctor.ps1

Output esperado:

XForge Engineer Doctor
======================
[1/8] Required structure
[OK]    AGENTS.md
[OK]    kilo.jsonc
[OK]    .kilo/commands
[OK]    .kilo/agents
[OK]    .kilo/skills
[OK]    .kilo/rules
[OK]    .xforge
...
Gate 4: Manual canonico DR-0181 (check-manual.ps1)...
  0 errors, 0 warnings, 24 OKs
Gate 5: Manual content (DR-0182)...
  0 errors, 0 warnings
Summary: 0 errors, 1 warning

Edge case: Se Gate 4 ou 5 falhar, commit sera bloqueado. Resolva antes de tentar commit.

Cenario 2: Auto-fix registries stale (B-089)

Contexto: Voce adicionou 5 skills novas. O registry ainda mostra o numero antigo. Re-gera do filesystem.

regenerate-registries.ps1

Output esperado:

Regenerating .kilo/core/registries/skill-registry.json from .kilo/skills...
  Found 176 skill entries  (era 171)
  Written (UTF-8 NoBOM)
...
Summary: skills 176, agents 81, rules 44, commands 144 (with publicCommands preserved)

Edge case: command-registry preserva publicCommands automaticamente. knowledge INDEX.json NAO e regenerado (schema complexo - manual regen via skills/knowledge-team).

Cenario 3: CI/CD pipeline (GitHub Actions)

Contexto: Voce quer rodar quality gates no GitHub Actions antes de merge.

# .github/workflows/quality.yml
name: XForge Quality Gates
on: [push, pull_request]
jobs:
  doctor:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with: { dotnet-version: '8.0.x' }
      - uses: actions/setup-python@v5
        with: { python-version: '3.12' }
      - name: Run doctor
        run: pwsh ./.kilo/automation/scripts/doctor.ps1 --json > doctor.json
      - name: Upload result
        uses: actions/upload-artifact@v4
        with: { name: doctor, path: doctor.json }

Output esperado (CI):

Run pwsh ./.kilo/automation/scripts/doctor.ps1 --json > doctor.json
{"summary":{"errors":0,"warnings":1,"oks":156},"gates":[...]}

Edge case: Em Linux/Mac, use pwsh (PowerShell Core). Se a plataforma nao suportar PS Core, instale via brew install --cask powershell ou apt install powershell.

Cenario 4: Gate 5 (B-090 content depth) com 1 pagina faltando secao

Contexto: Voce adicionou pagina nova mas esqueceu a secao Troubleshooting. Validator pega.

check-manual-content.ps1

Output esperado:

[2/3] Validating each page...
  [ERROR] docs/manual/12-xforge-init.html (12.3 KB) - troubleshooting too short (h3 count: 1)
  [ERROR] docs/manual/14-rag-system.html (15.8 KB) - size=15.8 KB (max 15)

[3/3] Summary
  Errors  : 2
  Warnings: 0
  OKs    : 10

FAIL - manual content nao atende B-090 (DR-0182)

Edge case: Validator nao bloqueia commit (Gate 5 so roda no doctor). Pre-commit so roda Gate 4. Use doctor.ps1 explicitamente antes de commit para validar Gate 5 tambem.

Cenario 5: Bypass pre-commit hook (emergencias)

Causa: Voce PRECISA commitar agora (hotfix de producao) mas algum gate falhou. Use bypass conscientemente.

git commit --no-verify -m "hotfix: critical security patch"
# Pre-commit hook NAO roda.
# Bypass fica no git log (auditable).

Output esperado (apos bypass):

[master abc1234] hotfix: critical security patch
# Commit feito, mas doctor.ps1 DEVE ser rodado separadamente
# para validar manualmente que esta tudo OK

Edge case: Bypass e' um anti-pattern. Use so em emergencias reais. Documente o motivo em CHANGELOG. Nunca use --no-verify para commits de feature normais.

Troubleshooting

"Encoding signals: Potential mojibake found"

Causa: 1 arquivo tem UTF-8 malformado (em-dash, smart quotes, etc).

Solucao: git diff para encontrar o arquivo. Substituir caracteres Unicode problematicos por ASCII (em-dash -> - , smart quotes -> "). Salvar como UTF-8 NoBOM.

"UTF-8 BOM found in registry"

Causa: Algum registry foi salvo com BOM (PowerShell Set-Content -Encoding UTF8).

Solucao: Re-salvar sem BOM. Use regenerate-registries.ps1 (ja escreve sem BOM via [System.IO.File]::WriteAllText com UTF8Encoding $false).

"RAG stale > 100 documents"

Causa: 100+ docs modificados entre reindex e pre-push.

Solucao: python .xforge/scripts/rag_local.py reindex antes do push. Para CI, adicione step de reindex antes do pre-push check.

"Loop discipline not 6/6 PASS"

Causa: Algum dos 6 criterios do loop-discipline.md falhou: build verde, test count, HTTP roundtrip, DB path, migrations, ou DR documenta coverage.

Solucao: Veja .kilo/rules/loop-discipline.md secao 4 (self-check). Resolva cada item antes de commitar.
