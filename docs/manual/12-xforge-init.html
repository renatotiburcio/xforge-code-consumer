12 - xforge-init: 5 Cenarios de Uso

O que e

xforge-init e' o comando de inicializacao que adapta o template XForge a um projeto novo ou existente. Diferente de git init (que apenas versiona) ou npm init (que cria package.json), o xforge-init instala 275 arquivos operacionais (skills, agents, commands, rules) + estrutura de governanca (.xforge/) + detecta stack + absorve documentacao pre-existente + gera DNA do projeto.

Localizacao: .kilo/automation/scripts/xforge-init.ps1 (Windows) e .kilo/automation/scripts/xforge-init.sh (Linux/Mac via PowerShell Core). Invocado como slash command /xforge-init no XForgeCode.

Quando usar

  • Quando voce tem um projeto novo (pasta vazia ou com pouco conteudo) e quer comecar com template
  • Quando voce tem um projeto existente e quer ganhar governanca XForge sem perder o que ja existe
  • Quando voce tem documentacao dispersa (pasta com specs, RFCs, decisoes) e quer absorver tudo no template
  • Quando voce quer duplicar o template em outro local sem perder o original
  • NAO use para atualizar o template em projeto ja inicializado (use /xforge-upgrade)
  • NAO use para remover template (nao ha comando uninit; apague .kilo/ e .xforge/ manualmente)

Como usar

Sintaxe canonica:

xforge-init [opcoes]

Comando interativo, mas flags abaixo cobrem cenarios comuns. Sem flags = cenario 1 (projeto limpo). Para abrir interface grafica, use /xforge-init dentro do XForgeCode.

# Windows PowerShell
.\.kilo\automation\scripts\xforge-init.ps1

# Linux/Mac (PowerShell Core)
pwsh ./.kilo/automation/scripts/xforge-init.ps1

# Com flags
xforge-init --absorb-docs ./legado/ --analyze-existing --template enterprise-v2

Parametros

ParametroTipoDefaultDescricao
--absorb-docs <path>stringvazioCaminho para pasta com documentacao pre-existente. Init le, analisa, memoriza, gera docs canonicas. Pasta pode ser deletada depois.
--analyze-existingboolfalseDetecta stack, convencoes, design do projeto existente. Init aprende em vez de sobrescrever.
--template <name>stringdefaultTemplate variante: default, enterprise-v2, startup-minimal, fiscal-br.
--config <file>stringxforge.yamlArquivo de configuracao customizado com variaveis de ambiente e stack.
--consumerboolfalseModo consumer: aplica reset de memoria/decisoes/audit (DR-0180 Fase 0). Para clonar template em projeto novo sem herdar historico.
--dry-runboolfalseMostra o que faria sem executar. Use para preview antes de commit.
--forceboolfalseSobrescreve arquivos existentes sem perguntar. Perigoso: use so se voce tem backup.
--quietboolfalseSuprime output. Retorna exit code apenas (0=ok, 1=erro). Para CI/CD.
--verboseboolfalseOutput detalhado de cada arquivo criado. Para debug.

Exemplos praticos

Cenario 1: Projeto Limpo (pasta vazia)

Contexto: Voce acabou de criar uma pasta para um projeto novo. Quer comecar do zero com template XForge.

mkdir meu-novo-projeto
cd meu-novo-projeto
xforge-init
# ou: pwsh ./.kilo/automation/scripts/xforge-init.ps1

Output esperado:

[xforge-init] v50.1.0 detectado
[detect-stack] Sem stack detectado (pasta vazia)
[init] Criando .kilo/ (skills, agents, commands, rules, automation)
[init] Criando .xforge/ (decisions, knowledge, memory, scripts)
[init] Criando docs/ (manual canonico)
[init] Criando .xforgecodeignore, AGENTS.md, README.md, kilo.jsonc
[init] Criando .gitignore + hooks pre-commit/pre-push
[init] Atualizando INDEX.json, RAG manifest
[done] Projeto inicializado. Proximo: git add . && git commit -m "feat: bootstrap xforge v50.1.0"

Edge case: Se voce esquecer de cd na pasta certa, init detecta diretorio pai e pergunta. Se voce ja tem arquivos na pasta, init pula (sem --force).

Cenario 2: Projeto com Pasta de Documentos (absorver)

Contexto: Voce tem uma pasta ./legado-docs/ cheia de specs, RFCs, decisoes previas. Quer absorver tudo no template, gerar docs canonicas no padrao XForge, e poder deletar ./legado-docs/ depois.

# Estrutura inicial
meu-projeto/
  src/
  legado-docs/
    spec-v1.md
    rfc-001-arquitetura.md
    decisoes-2025.md
    notas-reuniao.md
# Comando
xforge-init --absorb-docs ./legado-docs/

Output esperado:

[xforge-init] v50.1.0
[absorb-docs] 47 arquivos encontrados em ./legado-docs/
[analyze] Stack detectado: .NET 8 (de spec-v1.md)
[analyze] Decisoes extraidas: 12 (rfc-001, decisoes-2025, etc)
[generate] Criando .xforge/decisions/DR-0190-spec-original.md (absorvido de spec-v1.md)
[generate] Criando .xforge/decisions/DR-0191-rfc-arquitetura.md
[generate] Criando 18 knowledge entries em .xforge/knowledge/from-legado/
[generate] Atualizando .xforge/project-dna/PROJECT-DNA.md com DNA do legado
[done] 47 docs absorvidas, 12 DRs geradas, 18 knowledge entries
       Voce pode deletar ./legado-docs/ agora: rm -rf ./legado-docs/

Edge case: Se legado-docs/ tem 100+ arquivos, init pode demorar 2-5 minutos. Use --quiet em CI. Se algum doc nao parseavel (PDF binario, imagem), init pula com warning.

Cenario 3: Projeto Existente (sem pasta de docs)

Contexto: Voce tem um projeto ja em andamento com codigo, git, config, identidade visual. Quer ganhar governanca XForge sem perder nada. --analyze-existing faz init aprender em vez de sobrescrever.

cd projeto-legado  # ja tem package.json, src/, README.md
xforge-init --analyze-existing

Output esperado:

[xforge-init] v50.1.0
[analyze-existing] Detectando stack: Node.js 20 + TypeScript
[analyze-existing] Encontrados 53 arquivos em src/
[analyze-existing] Convencoes detectadas: ESLint standard, Prettier, Jest
[analyze-existing] Identidade visual: tailwind.config.js, cores #3B82F6
[init] Criando .kilo/ (275 files, NAO sobrescreve src/)
[init] Criando .xforge/ (NAO toca arquivos existentes)
[init] Criando .xforgecodeignore (exclui node_modules, dist, .env)
[init] Criando AGENTS.md customizado para o stack detectado
[init] Criando docs/manual/ (12 paginas canonicas)
[init] Atualizando .xforge/project-dna/PROJECT-DNA.md
[done] Stack: Node 20 + TypeScript. 53 src/ files preservados.
       Proximo: revisar AGENTS.md gerado e ajustar.

Edge case: Se o projeto tem .kilo/ ja existente (foi reinicializado?), init aborta com erro. Use --force se tiver certeza. Se src/ tem 1000+ arquivos, init demora. Use --verbose para acompanhar.

Cenario 4: Projeto Existente + Documentos (combinacao 2+3)

Contexto: Pior cenario - projeto legado grande com codigo E documentacao dispersa. Init faz deteccao, absorcao, e geracao em sequencia.

cd projeto-grande
xforge-init --analyze-existing --absorb-docs ./docs-antigos/ --verbose

Output esperado:

[xforge-init] v50.1.0
[analyze-existing] Stack: .NET 8 (ASP.NET Core + Blazor Server)
[analyze-existing] 312 arquivos em src/, 89 em tests/
[absorb-docs] 156 arquivos em ./docs-antigos/
[analyze] Padroes identificados: CQRS + MediatR + EF Core
[analyze] Decisoes: 23 extraidas de docs-antigos/decisoes-2024.md
[generate] 23 DRs geradas em .xforge/decisions/
[generate] 45 knowledge entries em .xforge/knowledge/from-antigos/
[generate] 1 DR-0192-cqrs-original.md (do legado)
[generate] 1 DR-0193-ef-core-decisoes.md
[init] .kilo/ criado sem sobrescrever
[init] AGENTS.md customizado para .NET 8
[done] Total: 312 src + 89 tests preservados, 156 docs absorvidas,
       23 DRs + 45 knowledge entries geradas.
       Recomendado: rm -rf ./docs-antigos/ (apos revisar DRs).

Edge case: Pode gerar 30+ DRs em projeto com muito legado. Cada DR fica em .xforge/decisions/. Revise antes de mergear. Se algum doc tiver conflito com decisao canonica XForge, init registra como DR-0194-conflito-pendente.

Cenario 5: Customizacao Avancada (template + config)

Contexto: Voce quer usar um template variante (ex: fiscal-br com regras LGPD pre-configuradas) e configurar com variaveis customizadas.

# xforge.yaml (pre-criado)
stack: dotnet
version: 8.0
lgpd:
  enabled: true
  dpo: renato@empresa.com.br
fiscal:
  uf: SP
  regime: real
  certificado: a1-path

# Comando
xforge-init --template fiscal-br --config ./xforge.yaml

Output esperado:

[xforge-init] v50.1.0
[template] Carregando fiscal-br (regime real + LGPD)
[config] Lendo xforge.yaml
[init] Stack: .NET 8.0
[init] LGPD habilitado, DPO renato@empresa.com.br
[init] Fiscal: SP, regime real, certificado A1
[init] Criando .kilo/skills/fiscal-tax-expert/ (com UF=SP pre-config)
[init] Criando .kilo/skills/lgpd-expert/
[init] Criando .xforge/knowledge/dominios/fiscal/sp/ (regras locais)
[init] Criando .kilo/rules/active-project-root-rules.md customizado
[done] Template fiscal-br aplicado. 3 skills adicionais instaladas.
       Proximo: revisar .kilo/rules/active-project-root-rules.md

Edge case: Se --template nao existe, init lista templates disponiveis. Se xforge.yaml tem schema invalido, init aborta com mensagem clara. Variaveis desconhecidas sao ignoradas com warning.

Troubleshooting

"xforge-init.ps1 nao encontrado"

Causa: Voce esta em uma pasta sem template XForge. Init nao roda sem template.

Solucao: cd para a pasta que contem .kilo/. Se voce quer criar um projeto novo do zero, faca git clone https://github.com/renatotiburcio/xforge primeiro, depois cp -r para pasta desejada, e so entao rode init.

"Permission denied" no Windows

Causa: ExecutionPolicy do PowerShell bloqueia scripts.

Solucao: Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned (one-time). Depois rode init normalmente.

"Stack detection returned null"

Causa: Nenhum sinal de stack encontrado (*.csproj, package.json, pyproject.toml, etc). Provavelmente pasta vazia ou projeto hibrido sem manifesto.

Solucao: Se for cenario 1 (limpo), e' normal - init usa defaults. Se for cenario 3 (existente) e voce tem codigo mas sem manifesto, crie um manifesto (mesmo package.json minimo) e rode init novamente.

"Init abortou mid-way, projeto quebrado"

Causa: Erro durante absorcao de docs ou deteccao de stack. Init nao tem rollback automatico.

Solucao: git status para ver o que foi commitado (nada deve estar). Se voce tinha backup de pre-init, restaure. Senao: delete .kilo/ e .xforge/ e rode init novamente. Use --dry-run antes em proxima vez para validar.

"DRs geradas estao erradas / incompletas"

Causa: Documentos absorvidos tinham formato nao-padrao (markdown mal-formado, plain text sem estrutura).

Solucao: DRs geradas tem source: "absorbed from <path>" no frontmatter. Edite manualmente cada DR-019X, corrija o problema, e mova para approved. Use a skill documentation-governor para ajuda na reescrita.
