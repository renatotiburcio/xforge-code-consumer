Wizard /release - Preparar Release

O que e

/release orquestra o processo de release: bump de versao (SemVer), geracao de changelog (commits desde ultima tag), validacao de tests, criacao de tag git, plano de rollback, e publicacao opcional. Integra com GitHub Releases.

Quando usar

  • Quando vai publicar versao nova em producao
  • Quando precisa gerar changelog automatico
  • Quando quer plano de rollback documentado antes do deploy

Como usar

Sintaxe canonica:

/release --bump patch

Para invocar: abra o XForge Code CLI e digite o comando. O wizard guia via Q&A interativo.

Parametros

ParametroTipoDefaultDescricaoObrigatorio?
--bumpstringobrigatoriomajor, minor, patchSim
--fromstringlast-tagTag de origemNao
--tostringHEADTag de destinoNao
--formatstringmarkdownmarkdown, json, htmlNao
--pushboolfalsePush automatico para remoteNao
--dry-runboolfalseMostra plano sem executarNao

Exemplos praticos

Cenario 1: Patch release (bug fix)

Contexto: Bug fix pequeno, vai publicar v50.0.3.

/release --bump patch

Output esperado:

[release] v50.0.1 -> v50.0.2
[release] CHANGELOG.md atualizado (3 commits desde v50.0.1):
  - fix: corrigir validacao de email em CreateCustomerCommand
  - fix: ajustar timeout do Hangfire para 30s
  - docs: atualizar README com setup do Postgres
[release] Tag v50.0.2 criada
[release] Push pendente (use --push para enviar)

Edge case: Sem --push, tag fica local ate voce fazer push manual.

Cenario 2: Minor release com changelog HTML

Contexto: Feature nova, vai publicar v51.0.0.

/release --bump minor --format html --output ./RELEASE-NOTES.html

Output esperado:

[release] v50.0.2 -> v51.0.0
[release] 12 commits desde ultima tag:
  Features (5):
    - feat: adicionar modulo de pagamentos (Stripe)
    - feat: adicionar notificacoes real-time (SignalR)
    - feat: adicionar jobs background (Hangfire)
    - feat: adicionar export de dados LGPD
    - feat: adicionar 2FA TOTP
  Fixes (4):
    - fix: corrigir N+1 query em ListCustomers
    - fix: ajustar CORS para producao
    - fix: corrigir validacao de CPF
    - fix: ajustar timeout de conexao
  Docs (3):
    - docs: atualizar arquitetura C4
    - docs: adicionar guia de deploy
    - docs: atualizar CHANGELOG

[release] RELEASE-NOTES.html gerado
[release] Tag v51.0.0 criada

Edge case: Use --format html para publicacao em portal interno.

Cenario 3: Major release com plano de rollback

Contexto: Breaking change (migracao de banco), precisa de rollback documentado.

/release --bump major --rollback-plan ./rollback.md

Output esperado:

[release] v51.0.0 -> v52.0.0
[release] BREAKING CHANGES detectados:
  - Migracao: Customer.Address de string para ValueObject
  - API: /api/customers/{id} retorna DTO diferente
  - Banco: nova tabela CustomerAddresses

[release] Plano de rollback gerado: ./rollback.md
  1. Reverter migracao: dotnet ef database update v51
  2. Reverter tag: git checkout v51.0.0
  3. Redeploy: docker-compose up

[release] Tag v52.0.0 criada (com warning: BREAKING)

Edge case: Major releases exigem --rollback-plan explicito.
