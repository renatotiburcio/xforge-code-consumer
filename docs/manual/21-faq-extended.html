21 - FAQ Estendido (perguntas avancadas)

O que e

Este FAQ complementa o /11-faq.html (FAQ basico) com perguntas avancadas sobre: integracao com CI/CD, customizacao de regras, extensao de MCPs, automacao de release, performance em escala, e troubleshooting de problemas especificos. Para duvidas iniciais, comece pelo /11-faq.html.

Quando usar

  • Quando ja leu o FAQ basico e precisa de mais profundidade
  • Quando quer implementar integracao customizada (CI/CD, IDEs, hooks)
  • Quando precisa extender XForge com regras/skills/MCPs proprios
  • Quando esta desenhando arquitetura multi-projeto ou multi-team

Como usar

FAQ avancado organizado por topico. Cada pergunta tem: contexto (quando voce se depara com isso), resposta completa (1-3 paragrafos), e referencias (links para DRs/skills/regras relevantes).

Exemplos praticos

Cenario 1: Integrar XForge com GitHub Actions

Contexto: Voce quer validar setup em CI a cada PR.

# .github/workflows/ci.yml (snippet)
jobs:
  xforge-health:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: sudo apt-get install -y powershell
      - run: pwsh ./.kilo/automation/scripts/doctor.ps1 --json > out.json
      - uses: actions/upload-artifact@v4
        with: { name: doctor, path: out.json }

Output esperado:

Job green se doctor exit 0. Se falha, baixa artifact para investigar.

Edge case: Para GH Actions em Windows runner, use runs-on: windows-latest e choco install powershell-core

Cenario 2: Criar MCP custom para ferramenta interna

Contexto: Sua empresa tem API interna e quer expor ao agente.

# Crie .kilo/mcp/optional/-mcp/install.sh
mkdir -p .kilo/mcp/optional/-mcp
# Crie -mcp/install.sh com npm install -g @yourco/mcp-server
# Crie .kilo/mcp/optional/.json.example
# Crie .kilo/skills/-mcp/SKILL.md (6 secoes B-090)
# Crie .xforge/decisions/DR-XXXX--mcp.md

Output esperado:

MCP custom aparece apos /mcp-lifecycle install . Siga o padrao dos MCPs OPTIONAL existentes (memory, markitdown, linear, slack).

Edge case: NAO modifique MCPs NATIVE - use OPTIONAL pattern. Padrao: .kilo/mcp/optional// + .kilo/mcp/optional/.json.example.

Cenario 3: Criar regra customizada

Contexto: Sua equipe tem convencao que nao existe nas 46 regras atuais.

# 1. Crie .kilo/rules/.md com frontmatter YAML:
# ---
# id: my-rule
# priority: high
# applicabilityScope: ["*"]
# ---
# 2. Adicione entrada em .kilo/rules/00-xforge-rule-index.md
# 3. Crie DR em .xforge/decisions/DR-XXXX-my-rule.md
# 4. Commit. Regra ativa imediatamente.

Output esperado:

Regra carregada por todos os skills/agents que importarem o rules index.

Edge case: Regra com applicabilityScope restrito so carrega em projetos relevantes (DR-0180).

Cenario 4: Versionamento semantico de releases

Contexto: Voce quer implementar SemVer no projeto.

# 1. Use Conventional Commits: feat: ... / fix: ... / BREAKING CHANGE: ...
# 2. Tag: git tag -a vX.Y.Z -m desc
# 3. Push: git push origin --follow-tags
# 4. Release notes automatico: GH Actions com releaser

Output esperado:

Tags com vX.Y.Z formam historico navegavel. GH Actions gera release notes do changelog.

Edge case: DR-0087: CHANGELOG e imutavel. Nunca edite versoes passadas - adicione entrada no topo (Unreleased).

Cenario 5: Performance com RAG grande (10000+ docs)

Contexto: Seu RAG tem muitos docs e queries estao lentas.

# 1. Reindex incremental em vez de full:
python .xforge/scripts/rag_local.py reindex --incremental
# 2. Use chunks menores em INDEX.json (reduza chunkSize de 1024 para 512)
# 3. Limite results por query (topK=10 em vez de 50)
# 4. Use cache local (--cache flag)

Output esperado:

Queries 3-5x mais rapidas. Reindex incremental atualiza apenas docs modificados.

Edge case: Para 100k+ docs, considere migrar para vector DB (Qdrant, Weaviate). RAG local e bom ate ~10k docs.

Cenario 6: Multi-projeto com mesmo template

Contexto: Voce tem 5 projetos que usam o mesmo template XForge.

# 1. Faca fork do template para seu org
# 2. Em cada projeto: git remote add template 
# 3. Para pull updates: git fetch template && git merge template/main --allow-unrelated-histories
# 4. Para customizar: use applicabilityScope nas suas rules/skills

Output esperado:

Cada projeto pode ter rules customizadas sem conflitar com o template base.

Edge case: DR-0180 (memory namespace) garante isolamento: knowledge cross-project so com DR + human approval.

Troubleshooting

CI falha com cannot find .NET SDK

Causa: Runner nao tem .NET instalado

Solucao: Adicione actions/setup-dotnet@v4 antes do build step. Para Ollama, use Ollama GitHub Action ou container pre-built.

MCP custom funciona local mas nao em CI

Causa: Env vars ou paths diferentes

Solucao: Use ${env:VAR_NAME} no JSON para resolucao dinamica. Em CI, defina vars em repo Settings > Secrets > Actions.

Reindex incremental nao pega arquivos novos

Causa: Cache stale ou path mismatch

Solucao: Delete .xforge/rag/.cache/ e rode reindex full. Verifique que arquivos estao em paths listados em INDEX.json.

Regra custom nao carrega

Causa: Falta entrada no rules index

Solucao: Verifique .kilo/rules/00-xforge-rule-index.md tem referencia a sua regra. Frontmatter YAML deve ter id, priority, applicabilityScope.
