13 - Stack Detection (DR-0180)

O que e

Stack Detection e' o mecanismo pelo qual XForge identifica o stack de um projeto (.NET, Node, Python, Rust, Go, Angular, etc) atraves de sinais no filesystem. Uma vez detectado, knowledge entries com applicabilityScope incompativeis sao filtrados, e skills/agents/commands ajustam suas recomendacoes.

Implementado em .kilo/automation/scripts/stack-detector.ps1 (PowerShell) + .kilo/skills/stack-aware-context/SKILL.md (skill canonica). Ativado em: xforge-init, knowledge router, doctor.ps1, e no inicio de cada sessao XForgeCode.

Quando usar

  • Quando voce precisa saber o stack do projeto atual (stack-detector.ps1 --report)
  • Quando adiciona novo stack e quer ensinar o detector (DR-0180 extension)
  • Quando knowledge entries nao carregam corretamente (provavelmente applicabilityScope filtro)
  • Quando o projeto e' hibrido (multi-stack) e precisa de deteccao por arquivo
  • NAO use para detectar framework dentro de um stack (use signal especifico, ex: Blazor WebApp vs Blazor Server)

Como usar

Sintaxe canonica:

.\.kilo\automation\scripts\stack-detector.ps1 [opcoes]

Para uso interativo via XForgeCode, basta abrir o projeto â€” a skill stack-aware-context carrega automaticamente.

# Detectar stack do projeto atual
stack-detector.ps1

# Gerar relatorio detalhado
stack-detector.ps1 --report

# Forcar deteccao em path especifico
stack-detector.ps1 --path C:/outro/projeto

# Modo JSON para pipelines CI
stack-detector.ps1 --json

# Validar entry de knowledge contra stack ativo
stack-detector.ps1 --check-scope .xforge/knowledge/INDEX.json

Parametros

ParametroTipoDefaultDescricao
--path <dir>stringPWDCaminho do projeto a analisar. Default = diretorio atual.
--reportboolfalseGera relatorio markdown com todos os sinais detectados, nao so o stack vencedor.
--jsonboolfalseOutput em JSON para pipelines CI/CD.
--check-scope <file>stringvazioValida todas entries de um INDEX.json contra stack ativo. Reporta incompativeis.
--min-confidence <0-100>int60Confianca minima para considerar stack detectado (0=sempre, 100=perfeito).
--multiboolfalseDetecta TODOS os stacks presentes (ex: .NET 8 + Node 20 em monorepo).
--update-promptboolfalseAtualiza .kilo/.stack-context.json com o stack detectado (cacheia para proxima sessao).

Exemplos praticos

Cenario 1: Detectar stack simples (.NET)

Contexto: Pasta com *.csproj e estrutura ASP.NET. Quer confirmar stack antes de comecar.

cd meu-projeto-dotnet
stack-detector.ps1

Output esperado:

[stack-detector] v1.0.0
[scan] 17 arquivos analisados em 234ms
[signals] csproj=3, sln=1, Program.cs=2, appsettings.json=2
[detect] Stack: dotnet (confidence=92%)
[detect] Sub-stack: ASP.NET Core 8.0 (LTS)
[detect] ORM provavel: Entity Framework Core 8
[done] .kilo/.stack-context.json atualizado

Edge case: Se confidence < 60%, detector pergunta "multi-stack?" e lista todos os candidatos. Se voce tem global.json com version, usa isso para confirmar versao exata.

Cenario 2: Multi-stack (monorepo)

Contexto: Repo tem frontend React + backend Node + shared types TypeScript. Stack detection multi-stack.

cd monorepo
stack-detector.ps1 --multi

Output esperado:

[stack-detector] v1.0.0
[scan] 487 arquivos analisados
[signals] package.json=8, tsconfig.json=12, Cargo.toml=1, go.mod=2
[detect] Stack primario: node (confidence=78%)
[detect] Sub-stacks: react (frontend), typescript (universal), rust (tooling)
[detect] Nao detectado: go (signal isolado em /tools)
[done] Monorepo detectado. Knowledge routing ajustara por diretorio.

Edge case: Se voce tem package.json + Cargo.toml na raiz mas src/ so tem Rust, detector usa densidade de arquivos por stack. Use --path src/ para override.

Cenario 3: Projeto sem stack obvio

Contexto: Pasta com docs e configs mas sem codigo (ex: monorepo em fase de planejamento).

cd docs-project
stack-detector.ps1

Output esperado:

[stack-detector] v1.0.0
[scan] 89 arquivos analisados
[signals] Nenhum manifesto de stack detectado
[detect] Stack: unknown (confidence=0)
[warning] Recomendacao: especifique stack com 'xforge init --template <name>'
[warning] Ou crie manifesto (package.json, *.csproj) para deteccao automatica
[done] Stack context nao salvo (insuficiente confidence)

Edge case: Se voce tem README.md com mencoes a stack (ex: "Node.js + PostgreSQL"), detector nao parseia texto livre. Workaround: criar .stack-overrides.json na raiz com stack explicito.

Cenario 4: Validar knowledge entries contra stack

Contexto: voce tem 319 knowledge entries. Quer saber quantas serao carregadas em um projeto .NET 8.

stack-detector.ps1 --check-scope .xforge/knowledge/INDEX.json

Output esperado:

[stack-detector] v1.0.0
[detect] Stack ativo: dotnet
[scope-check] Varrendo 319 entries em .xforge/knowledge/INDEX.json
[scope-check] Compativeis: 187 (58.7%)
[scope-check] Incompativeis: 132 (41.3%)  -- serao filtradas
[scope-check] Breakdown incompativeis:
  - python: 67
  - node: 41
  - react: 18
  - rust: 4
  - go: 2
[done] Use --filter para ver quais entries incompativeis serao excluidas

Edge case: Entries com applicabilityScope: ["*"] sao sempre compativeis (universais). Para adicionar uma entry sua, edite o INDEX.json e adicione applicabilityScope no formato ["dotnet", "python", "*"] para multi-stack.

Cenario 5: Adicionar novo stack (DR-0180 extension)

Contexto: Voce quer que XForge reconheca Elixir (nao vem por default). Adiciona sinal customizado.

# Editar .kilo/automation/scripts/stack-detector.ps1
$stackSignals = @{
  "elixir" = @("mix.exs", "mix.lock", "lib/**/*.ex", "lib/**/*.exs")
  # outros...
}

# Adicionar entry em .xforge/core/registries/stack-registry.json (criar se nao existir)
{
  "id": "elixir",
  "name": "Elixir",
  "signals": ["mix.exs", "mix.lock"],
  "applicabilityScope": ["elixir"],
  "version": ">= 1.14"
}

Output esperado (apos re-executar detector):

[stack-detector] Stack customizado carregado: elixir
[detect] Stack: elixir (confidence=85%)
[done] .kilo/.stack-context.json atualizado com elixir

Edge case: Stack customizado nao aparece no help do detector ate ser registrado formalmente. Crie .kilo/core/registries/stack-registry.json para autodiscovery. Adicionar em applicabilityScope das knowledge entries que devem carregar.

Troubleshooting

Detector retorna "unknown" mas sei meu stack

Causa: Manifesto na raiz, mas codigo em subpasta. Detector so olha raiz por default.

Solucao: stack-detector.ps1 --path ./backend/ para escanear subpasta. Ou criar symlink/copy do manifesto na raiz.

Knowledge entry nao carrega mesmo com applicabilityScope correto

Causa: Cache de .kilo/.stack-context.json esta stale.

Solucao: stack-detector.ps1 --update-prompt ou delete .kilo/.stack-context.json e re-abra o projeto. Cache expira em 24h por default.

"Multi-stack detected: not supported"

Causa: Voce passou --multi mas detector achou 1 stack so.

Solucao: Sem --multi o detector retorna o stack primario. Use --multi explicitamente para forcar analise multi-stack. Se o stack primario estiver errado, forneca --path para o sub-diretorio relevante.

"Stack registry not found"

Causa: .kilo/core/registries/stack-registry.json nao existe (template-only ou deletado).

Solucao: git checkout main -- .kilo/core/registries/stack-registry.json ou rode regenerate-registries.ps1 que cria o arquivo base. Para customizar, edite apos regenerar.
