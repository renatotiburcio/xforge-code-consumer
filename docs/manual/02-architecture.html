02 - Arquitetura (C4 L1/L2/L3)

O que e

XForge tem 2 diretorios canonicos: .kilo/ (operacional, versionado, 275+ arquivos) e .xforge/ (estado do projeto, decisoes + knowledge + memory). Diferente de monorepo, aqui e' template + runtime: .kilo/ nao muda por projeto (e' o template), .xforge/ evolui com cada clone.

Quando usar

  • Quando voce quer entender ONDE colocar novos arquivos (skills, agents, DRs, knowledge)
  • Quando voce precisa diferenciar template-only de project-state (DR-0180)
  • Quando vai customizar e quer saber o que customizar (resposta: .xforge/, NAO .kilo/)
  • Quando vai clonar o template (use --consumer para resetar .xforge/)

Como usar

Sintaxe canonica (C4):

# C4 L1: Containers
[VS Code + XForgeCode] -> [.kilo/ + .xforge/ + docs/]

# C4 L2: .kilo/ components
.kilo/
  skills/         (172 modulos)
  agents/         (53 root + 28 genius)
  commands/       (144 slash)
  rules/          (41 behavioral)
  core/registries/ (17 JSONs)
  automation/     (90+ scripts)

# C4 L3: .xforge/ components
.xforge/
  decisions/      (95 ativos + 52 archive)
  knowledge/      (319 entries)
  memory/         (16 files)
  scripts/        (15 PowerShell)
  templates/      (6 templates)
  rag/            (config + indexes + chunks)
  reports/        (audits + quality)
  version.json    (50.1.0, codename)

Parametros

ParametroTipoDefaultDescricao
--kilo-pathstring./.kiloOverride: .kilo/ em outro local (multi-root workspaces).
--xforge-pathstring./.xforgeOverride: .xforge/ em outro local.
--template-onlyflagfalseLista so arquivos template-only (excluidos em clone consumer).
--project-stateflagfalseLista so arquivos de estado do projeto.

Exemplos praticos

Cenario 1: Mapear diretorios (primeira inspecao)

Contexto: Voce acabou de clonar o template. Quer ver o que tem.

cd xforge
tree -L 2 -I 'node_modules|.git'
# Ou no Windows:
Get-ChildItem -Recurse -Depth 2 | Where-Object { -not $_.PSIsContainer } | Group-Object PSParentPath

Output esperado (resumido):

.
|-- .kilo/
|   |-- skills/      (172 arquivos)
|   |-- agents/      (53 + 28 arquivos)
|   |-- commands/    (144 arquivos)
|   |-- rules/       (41 arquivos)
|   |-- core/        (17 JSONs)
|   `-- automation/  (90+ scripts)
|-- .xforge/
|   |-- decisions/   (95 DRs ativos + 52 archive)
|   |-- knowledge/   (210 files / 319 entries)
|   |-- memory/      (16 files)
|   `-- scripts/     (15 scripts)
|-- docs/             (12 paginas canonicas)
|-- AGENTS.md         (entry point)
|-- ARCHITECTURE.md   (este arquivo espelho)
|-- CLAUDE.md         (Claude Code alias)
|-- README.md         (landing GitHub)
|-- CHANGELOG.md      (release history)
`-- STATUS.md         (status master)

Edge case: tree nao vem no Windows por default. Use Get-ChildItem -Recurse ou instale tree via Chocolatey.

Cenario 2: Listar template-only files (DR-0180)

Contexto: Voce quer saber o que sera EXCLUIDO quando voce clonar como consumer.

Get-Content .xforge/template-only.json | ConvertFrom-Json | Format-List

Output esperado:

templateOnlyPaths:
  - .xforge/decisions/archive/
  - .xforge/decisions/DR-NNNN-*.md
  - .xforge/sessions/
  - .xforge/audit/
  - .xforge/memory/sessions/
  - .xforge/learning/
  - .xforge/feedback/
  - .xforge/rag/indexes/
  - .xforge/rag/chunks/

# Quando voce roda: xforge-init --consumer
# Esses paths sao resetados/limpos
# O resto (decisions/ ativo, knowledge/, scripts/) e preservado

Edge case: template-only.json e o manifesto canonico. Edita-lo afeta clones futuros. Para excluir paths customizados, adicione ao array.

Cenario 3: Adicionar nova skill (template-side)

Contexto: Voce criou uma skill nova e quer adiciona-la ao template.

# Criar skill
mkdir -p .kilo/skills/minha-skill
cat > .kilo/skills/minha-skill/SKILL.md <<EOF
---
name: minha-skill
description: O que minha skill faz
type: skill
category: Minha Categoria
applicabilityScope: ["dotnet"]
---
# Conteudo...
EOF

# Regenerar registry
regenerate-registries.ps1

# Validar
doctor.ps1
git add .kilo/skills/minha-skill/
git commit -m "feat: add minha-skill"

Output esperado:

[regenerate] 173 skills (era 172)
[doctor] 0 errors
[git] feat: add minha-skill

Edge case: Se o registry nao for regenerado, a skill existe mas nao e indexada. Sempre rode regenerate-registries.ps1 apos criar/modificar componentes.

Cenario 4: Adicionar nova knowledge entry (project-side)

Contexto: Voce aprendeu algo novo do projeto e quer memorizar no project state.

# Criar arquivo
cat > .xforge/knowledge/projeto/especifico-projeto.md <<EOF
# Especifico deste projeto
Insight: ...
Decisao: ...
Refs: DR-XXXX
EOF

# Adicionar ao INDEX.json
{
  "path": ".xforge/knowledge/projeto/especifico-projeto.md",
  "domain": "projeto",
  "applicabilityScope": ["*"],
  "trustScore": 90,
  "keywords": ["projeto", "especifico"]
}

# Reindexar RAG
python .xforge/scripts/rag_local.py reindex

Output esperado:

[INDEX] 320 entries (era 319, +1)
[reindex] 748 docs (era 747, +1)
[done] Knowledge router recarregado

Edge case: Edicao do INDEX.json e manual. Para automatizar, use a skill knowledge-ingestion.

Cenario 5: Diferenciar template-only vs project-state (clone consumer)

Contexto: Voce quer usar o template em um novo projeto mas SEM herdar o historico (decisoes passadas, audit logs, learning).

# Clone normal (herda tudo)
git clone https://github.com/renatotiburcio/xforge.git meu-novo-projeto
cd meu-novo-projeto
# Tem .xforge/decisions/ com 95 DRs que NAO sao do seu projeto
# Tem .xforge/memory/sessions/ com sessoes de quem desenvolveu o template

# Reset para uso consumer (DR-0180 Fase 0)
xforge-init --consumer
# Remove: archive/, sessions/, audit/, learning/, feedback/, rag/indexes/
# Preserva: decisions/ ativo (mas zera), knowledge/, scripts/, memory/genius/

Output esperado:

[consumer-reset] Limpando template-only paths:
  - .xforge/decisions/archive/ (52 files deleted)
  - .xforge/sessions/ (3 files deleted)
  - .xforge/audit/ (15 files deleted)
  - .xforge/learning/ (8 files deleted)
  - .xforge/feedback/ (2 files deleted)
  - .xforge/rag/indexes/ (1 file deleted)
  - .xforge/rag/chunks/ (1 file deleted)
[done] Consumer reset completo. 82 files removed.

Edge case: --consumer apaga arquivos. Garanta que voce ja commitou o que importa. Para preservar decisoes, exporte antes.

Troubleshooting

"Conflito entre .kilo/ e customizacao do projeto"

Causa: Voce editou .kilo/ ao inves de .xforge/. Template-side e imutavel por projeto (so muda em commits do repo do template).

Solucao: Reverta edicao em .kilo/. Customizacoes vao em .xforge/ (decisions/, knowledge/, memory/, scripts/) ou em .kilo/.stack-context.json (per-projeto cache).

"RAG nao encontra knowledge nova"

Causa: RAG reindex nao foi rodado. Chunks.jsonl nao tem a nova entry.

Solucao: python .xforge/scripts/rag_local.py reindex. Verifique lastIndexedAt em .xforge/rag/manifest.json.

"Doctor reclama de registries stale"

Causa: Voce criou/modificou skills, agents, rules, commands, mas registries nao foram regenerados.

Solucao: regenerate-registries.ps1 (escreve 4 registries atomicamente).

"Onde fica X?" (regra geral)

Causa: Duvida canonica. Use a matrix:

Solucao:

  • SKILL nova (reutilizavel) -> .kilo/skills/nome/SKILL.md (template)
  • AGENT novo (AI entity) -> .kilo/agents/nome.md (template)
  • COMMAND novo (slash) -> .kilo/commands/nome.md (template)
  • RULE nova (behavioral) -> .kilo/rules/nome.md (template)
  • KNOWLEDGE entry nova -> .xforge/knowledge/<dominio>/nome.md + INDEX.json (project)
  • DR nova -> .xforge/decisions/DR-NNNN-titulo.md (project)
  • Script custom -> .xforge/scripts/nome.ps1 (project)
