16 - Knowledge applicabilityScope (DR-0180)

O que e

applicabilityScope e' o campo em cada knowledge entry que declara para quais stacks o conhecimento se aplica. Quando o projeto ativo e' .NET 8, knowledge entries com applicabilityScope: ["python"] sao filtradas. Diferente de busca por nome de arquivo, applicabilityScope e' uma tag semantica que o Knowledge Router usa para selecionar 187 de 319 entries (58.7%) em um projeto .NET medio.

Localizacao: .xforge/knowledge/INDEX.json (campo applicabilityScope em cada entry). Skill canonica: stack-aware-context. Validador: stack-detector.ps1 --check-scope.

Quando usar

  • Quando voce cria uma nova knowledge entry (declarar stack)
  • Quando uma knowledge entry aparece mesmo em projeto de stack diferente (scope mal declarado)
  • Quando voce quer auditar quais entries carregam em um projeto
  • Quando voce quer criar knowledge multi-stack (aplica em varios)
  • NAO use ["*"] indiscriminadamente (defeats the purpose)
  • NAO use para versionamento (use TTL/validity field separado, se implementado)

Como usar

Sintaxe canonica (em cada entry do INDEX.json):

{
  "path": ".xforge/knowledge/dominios/fiscal/nfe.md",
  "domain": "fiscal",
  "applicabilityScope": ["dotnet", "python"],  // Universal: ["*"]
  "trustScore": 90,
  "keywords": ["nfe", "sefaz", "xml"]
}

Validacao via skill canonica stack-aware-context ou comando:

# Ver quantas entries carregam no projeto atual
stack-detector.ps1 --check-scope .xforge/knowledge/INDEX.json

# Forcar aplicabilidade universal (use com cuidado)
# Editar .xforge/knowledge/INDEX.json manualmente

Exemplos praticos

Cenario 1: Knowledge universal (LGPD)

Contexto: LGPD aplica a QUALQUER stack (BR tem a lei). Use ["*"].

{
  "path": ".xforge/knowledge/compliance/lgpd-basics.md",
  "domain": "lgpd",
  "applicabilityScope": ["*"],
  "trustScore": 95,
  "source": "LGPL Lei 13.709/2018 + ANPD guidelines"
}

Output esperado (em qualquer projeto):

[scope-check] LGPD entry: aplica a TODOS os stacks
[scope-check] Sera carregada em qualquer projeto

Edge case: ["*"] so para verdadeiramente universal. Duvida? Escolha 2-3 stacks especificos em vez de "*".

Cenario 2: Knowledge .NET only (AutoMapper)

Contexto: AutoMapper e' so .NET. Skill canonica XForge exige AutoMapper sobre Mapster.

{
  "path": ".xforge/knowledge/packs/xforge-automapper/profiles.md",
  "domain": "pack",
  "applicabilityScope": ["dotnet"],
  "trustScore": 88,
  "version": ">= 12.0"
}

Output esperado (em projeto Python):

[scope-check] .NET entry: nao carrega em projeto Python
[scope-check] Sera filtrada

Edge case: Se voce tem AutoMapper em projeto Python (improvavel mas possivel via wrappers), escopo errado. Use ["dotnet", "python"] ou crie entry especifica.

Cenario 3: Knowledge multi-stack (REST APIs)

Contexto: Conceitos de REST API aplicam a .NET, Node, Python, Go. Multi-stack.

{
  "path": ".xforge/knowledge/padroes/rest-apis.md",
  "domain": "padroes",
  "applicabilityScope": ["dotnet", "node", "python", "go"],
  "trustScore": 90,
  "keywords": ["rest", "http", "api"]
}

Output esperado (em projeto Java):

[scope-check] Java nao esta no scope (dotnet, node, python, go)
[scope-check] Sera filtrada (mas a entry e' util para Java tambem!)

Edge case: Se Java tambem usa REST, adicione "java" ao scope. Trade-off: scope mais restritivo = mais precisa, menos inclusivo.

Cenario 4: Knowledge dominio-especifico (NFe Brasil)

Contexto: NFe e' exclusivamente BR. Aplica a qualquer stack usado em software BR, mas NAO a linguagens sem implementacao.

{
  "path": ".xforge/knowledge/dominios/fiscal/nfe-basics.md",
  "domain": "fiscal",
  "applicabilityScope": ["fiscal"],  // NAO misturar com dotnet/python
  "trustScore": 95,
  "keywords": ["nfe", "sefaz", "xml", "brasil"]
}

Output esperado (em projeto qualquer):

[scope-check] Scope "fiscal" aplica quando projeto esta em dominio fiscal
[scope-check] Em projeto ERP BR: carrega
[scope-check] Em projeto blog pessoal: NAO carrega

Edge case: Para que scope "fiscal" seja reconhecido, o projeto precisa de um sinal de dominio. Skill stack-aware-context detecta atraves de keywords em arquivos ou config customizada.

Cenario 5: Adicionar nova knowledge entry (com scope correto)

Contexto: Voce escreveu .xforge/knowledge/padroes/feature-flags.md. Adiciona ao INDEX.json com scope apropriado.

# 1. Criar arquivo
mkdir -p .xforge/knowledge/padroes
cat > .xforge/knowledge/padroes/feature-flags.md <<EOF
# Feature Flags
Feature flags permitem toggling de funcionalidades sem deploy...
EOF

# 2. Adicionar ao INDEX.json
{
  "path": ".xforge/knowledge/padroes/feature-flags.md",
  "domain": "padroes",
  "applicabilityScope": ["dotnet", "node", "python", "rust", "go", "react"],
  "trustScore": 85,
  "keywords": ["feature-flags", "toggle", "deploy"]
}

# 3. Validar
stack-detector.ps1 --check-scope .xforge/knowledge/INDEX.json

Output esperado:

[scope-check] 320 entries (era 319, +1)
[scope-check] feature-flags.md: scope=6 stacks, trustScore=85
[scope-check] Sera carregada em 6 tipos de stack

Edge case: Se voce esquecer de adicionar ao INDEX.json, a entry existe no filesystem mas nao e indexada. RAG reindex adiciona ao chunks.jsonl mas nao ao knowledge router. Sempre atualize ambos.

Troubleshooting

Knowledge entry nao carrega mesmo com scope correto

Causa: Stack detection retornou "unknown" (sem manifestos detectados). Router nao sabe qual scope aplicar.

Solucao: stack-detector.ps1 --update-prompt para forcar redeteccao. Crie manifesto (package.json, *.csproj) se pasta nao tem.

"applicabilityScope malformado"

Causa: Entry com scope escrito errado (ex: ["DotNet"] em vez de ["dotnet"]). Router case-sensitive.

Solucao: Use sempre lowercase. Lista canonica: dotnet, python, node, react, angular, vue, rust, go, java, html-tailwind, fiscal, contabil, trabalhista, lgpd, *.

"Entry com scope vazio []"

Causa: Entry criada sem scope declarado. Router assume "incompativel" e filtra.

Solucao: Adicione scope. Se realmente universal, use ["*"]. Duvida? Comece com 1 stack especifico.

Knowledge INDEX.json nao parseia

Causa: JSON corrompido (editor salvou mal, virgula faltando, etc).

Solucao: python -c "import json; json.load(open('.xforge/knowledge/INDEX.json'))" para localizar erro. Use editor com lint JSON (VS Code faz). Backup em INDEX.json.bak-YYYYMMDD-HHMM.
